# 测试覆盖策略（Test Coverage Strategy）

在需要补充测试、做重构前评估、或接到「给这段代码加测试」请求时，用本技能分析并给出覆盖策略。

## 架构约定（Vernify）

- **主后端**：Next.js（前端 + BFF）；**扩展服务**：FastAPI（`Web/backend/`）仅为 Next 提供 AI/LaTeX 等能力。扩展服务测试在 `Web/backend/tests/`。

## 何时使用

- 用户说「给 xxx 加测试」「这段代码没测试」
- 重构前评估「哪些地方需要测试保护」
- 新功能上线前检查「关键路径是否有测试」
- 接到「提高测试覆盖」任务时

## 执行步骤

### 1. 明确分析范围

- **目标**：单个模块、一条 API、一个组件、还是整个服务？
- **现状**：是否已有测试？在 `tests/`、`__tests__/`、`*.test.*` 中查找。
- **需求**：用户期望的覆盖程度（快速 smoke、核心路径、高覆盖）。

### 2. 识别关键行为

- **API**：哪些端点、方法、状态码、错误分支？
- **业务逻辑**：核心计算、校验、状态转换、边界条件？
- **UI**：关键交互、表单提交、错误展示？
- **依赖**：外部 API、数据库、环境变量，哪些需要 mock？

### 3. 选择测试层级

| 场景 | 推荐层级 | 理由 |
|------|----------|------|
| 纯函数、工具函数 | 单元 | 快速、稳定、易维护 |
| API 路由、服务层 | 集成 | 验证请求/响应、中间件、DB |
| 用户流程、多页交互 | E2E | 端到端验证 |
| 已有单元、缺集成 | 集成 | 补足接口层 |
| 复杂组件逻辑 | 单元 + 集成 | 逻辑单测 + 集成渲染 |

### 4. 制定优先级

- **P0**：核心业务、支付/登录/关键流程 → 优先
- **P1**：常用功能、错误处理 → 其次
- **P2**：边缘情况、性能、可观测性 → 视时间

### 5. 输出策略摘要

向用户或后续步骤提供：

- **缺口**：当前无覆盖的模块/端点/组件
- **建议**：为每项建议测试类型（单元/集成/E2E）、文件位置、示例用例名
- **Mock 需求**：需 mock 的外部依赖
- **命令**：如何运行相关测试（pytest、npm test 等）

## 示例

**用户**：「给 `/api/v1/quiz/submit` 加测试。」

**策略产出**：

- **现状**：无现成测试。
- **关键行为**：POST 请求、Zod 校验、Supabase 写入、返回提交结果。
- **建议**：
  - 集成测试（扩展服务 FastAPI）：`Web/backend/tests/integration/test_quiz_submit.py`
  - 用例：`test_submit_valid_answer_200`、`test_submit_invalid_schema_422`、`test_submit_unauthorized_401`
- **Mock**：Supabase client 或使用测试库。
- **命令**：`cd Web/backend && poetry run pytest tests/integration/test_quiz_submit.py -v`

## 与流程的衔接

- 策略制定后可委托 **test-writer** 编写具体测试。
- 若在 TDD 流程中，策略可作为 tdd-plan-and-implement 的输入，指导任务拆分。
